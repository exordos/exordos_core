# Руководство администратора

Это руководство предназначено для администраторов локальной инсталляции Exordos Core. Оно описывает подготовку хост-машины, развёртывание платформы, настройку доступа и базовые действия по эксплуатации.

## Область применения

Руководство описывает локальную установку на одной хост-машине под управлением Ubuntu Linux 24.04 или 26.04 с виртуализацией QEMU/KVM и libvirt. Такой контур предназначен для разработки, тестирования и функциональной проверки платформы.

## Предварительные требования

Для развёртывания требуются:

- хост-машина под управлением Ubuntu Linux 24.04 или 26.04;
- права для установки системных пакетов и настройки виртуализации: QEMU версии 8.2 или выше и libvirt версии 10.0 или выше;
- доступ к репозиторию Exordos для установки CLI и загрузки образа;
- SSH-ключ администратора для доступа к основному узлу;
- хост-машина с минимум 4 CPU, 8 ГБ оперативной памяти и 50 ГБ свободного дискового пространства для основной виртуальной машины и управляемых Node.

## Установка Exordos CLI

Установите Exordos CLI:

```bash
curl -fsSL https://repo.exordos.com/install.sh | sh
```

Проверьте доступные команды и параметры:

```bash
exordos --help
exordos bootstrap --help
```

## Подготовка хост-машины

Установите QEMU/KVM версии 8.2 или выше, libvirt версии 10.0 или выше и утилиты для работы с образами:

```bash
sudo apt update
sudo apt install qemu-kvm qemu-utils libvirt-daemon-system libvirt-dev mkisofs -y
```

Добавьте пользователя, от имени которого будет выполняться CLI, в группы `libvirt` и `kvm`:

```bash
sudo adduser $USER libvirt
sudo adduser $USER kvm
```

После изменения групп выйдите из системы и войдите повторно.

Инициализируйте хост-машину как гипервизор от имени суперпользователя:

```bash
sudo exordos compute hypervisors init
```

Просмотрите параметры инициализации:

```bash
exordos compute hypervisors init --help
```

## Развёртывание платформы

Выполните bootstrap Exordos Core:

```bash
exordos bootstrap -i <version-or-image> -f -m core --ssh-public-key /path/to/public/key
```

Аргумент `<version-or-image>` принимает путь к локальному образу или номер версии платформы из репозитория. Примеры:

```bash
exordos bootstrap -i /path/to/exordos-core.raw -m core
```

```bash
exordos bootstrap -i <version> -m core
```

Например, для установки версии `0.2.1`:

```bash
exordos bootstrap -i 0.2.1 -m core
```

Ключевые параметры bootstrap:

| Параметр | Описание |
| --- | --- |
| `--profile` | Профиль инсталляции: `develop`, `small`, `medium`, `large` или `legacy`. |
| `--cidr` | CIDR основной сети платформы. По умолчанию: `10.20.0.0/22`. |
| `--core-ip` | IP-адрес основной виртуальной машины. По умолчанию используется второй адрес из `--cidr`. |
| `--admin-password` | Пароль администратора для HTTP API. Если не задан, генерируется автоматически. |
| `--save-admin-password-file` | Сохраняет пароль администратора для HTTP API в файл. |
| `--ssh-public-key` | Путь к публичному SSH-ключу для доступа к основной виртуальной машине. |
| `--hyper-connection-uri` | URI подключения к удалённому гипервизору libvirt. |
| `--hyper-storage-pool` | Пул libvirt для дисков виртуальных машин. |

После успешного завершения `bootstrap` платформа создаёт и запускает основную виртуальную машину, а команда выводит логин и пароль администратора для HTTP API. Если указан `--save-admin-password-file`, пароль API сохраняется в заданный файл. При ошибке команда выводит сообщение об ошибке и завершается с ненулевым кодом.

Сохраните сгенерированные учётные данные API администратора в защищённом хранилище. Не передавайте пароль в журналах, манифестах или системах контроля версий.

### Учётные данные API администратора

Для каждой локальной инсталляции `bootstrap` создаёт отдельные логин и пароль администратора для доступа к HTTP API. Постоянные общие учётные данные не используются. Используйте логин и пароль, выведенные командой `bootstrap` или сохранённые с помощью `--save-admin-password-file`.

Доступ к основной виртуальной машине выполняется по SSH-ключу, публичная часть которого передана при bootstrap через параметр `--ssh-public-key`; логин и пароль API для SSH не используются.

Для воспроизводимого развёртывания можно передать пароль API явно:

```bash
exordos bootstrap -i <version> -m core --admin-password <admin-password>
```

Не размещайте фактические учётные данные API инсталляции в общедоступной документации.

## Настройка доступа

### SSH-доступ

Подключитесь к основной виртуальной машине. Для подключения используется закрытая часть SSH-ключа, публичная часть которого была передана через `--ssh-public-key` при bootstrap:

```bash
ssh ubuntu@<core-ip>
```

Для стандартной сети bootstrap `10.20.0.0/22` адрес основной виртуальной машины — `10.20.0.2`. Логин и пароль администратора для HTTP API не используются для SSH.

### Настройка CLI

Зарегистрируйте endpoint платформы и контекст администратора. Для параметров `<ADMIN_USERNAME>` и `<ADMIN_PASSWORD>` используйте учётные данные HTTP API, созданные командой `bootstrap`:

```bash
exordos settings set-realm local --endpoint http://<core-ip>:11010 --current
exordos settings set-context local --name admin -u <ADMIN_USERNAME> -p <ADMIN_PASSWORD> --current
```

После настройки CLI проверьте подключение:

```bash
exordos compute hypervisors list
exordos elements list
```

### Доступ к HTTP API

Для вызовов HTTP API используйте логин и пароль API администратора, созданные командой `bootstrap`, чтобы получить `access_token` в IAM-сервисе. Полный пример запроса приведён в [инструкции локального развёртывания](local_deployment.md#доступ-через-api). Передавайте полученный токен как Bearer-токен в последующих запросах.

## Проверка запуска и ответы команд

После настройки CLI выполните последовательную проверку доступности инсталляции:

| Команда | Назначение и ожидаемый результат |
| --- | --- |
| `exordos realms list` | Выводит список локальных realm. В списке должен присутствовать realm `local`. |
| `exordos compute hypervisors list` | Выводит зарегистрированные гипервизоры. В списке должен присутствовать подготовленный гипервизор. |
| `exordos compute nodes list` | Выводит список Node. После bootstrap в списке доступна основная Node платформы. |
| `exordos elements list` | Выводит список установленных элементов. |
| `exordos em resources list` | Выводит ресурсы элементов и их состояния. |

Команды `exordos elements list` и `exordos em resources list` поддерживают параметры `--filters key=value`, `--fields <field>`, `--output table|json|html|yaml`, `--watch` и `--interval <seconds>`. Формат вывода по умолчанию — таблица. При использовании `--watch` CLI повторно запрашивает данные с указанным интервалом.

## Базовая эксплуатация

### Контроль ресурсов

Для контроля состояния вычислительных ресурсов используйте:

```bash
exordos compute hypervisors list
exordos compute nodes list
```

Для просмотра ресурсов элементов используйте:

```bash
exordos em resources list
exordos em resources show <uuid>
```

Ресурс проходит состояния `NEW → IN_PROGRESS → ACTIVE`. Состояние `IN_PROGRESS` означает, что платформа продолжает приводить ресурс к целевому состоянию.

### Управление элементами

Для просмотра установленных элементов используйте:

```bash
exordos elements list
```

Установка элемента выполняется командой:

```bash
exordos elements install <element-name>
```

Описание манифестов и ресурсов элементов приведено в [справочнике по манифесту](../em/manifest.md).

### Конфигурации и секреты

Передавайте конфигурации и конфиденциальные данные через ресурсы Config и Secret. Не включайте пароли, ключи и сертификаты в открытом виде в манифесты, журналы команд или репозитории исходного кода.

## Диагностика

Если ресурс не выходит из `IN_PROGRESS`, сначала проверьте состояние ресурса, Node и гипервизора:

```bash
exordos em resources show <uuid>
exordos compute nodes list
exordos compute hypervisors list
```

Типовые проблемы с состояниями ресурсов, конфигурациями и bootstrap-скриптами описаны в разделе [Диагностика проблем](troubleshooting.md).

## Завершение работы инсталляции

Для штатного завершения локальной инсталляции удалите локальный realm:

```bash
exordos realms delete local
```

Команда удаляет локальный realm и завершает работу локального экземпляра. Используйте её только для завершения тестового или экспертного контура. Для нового запуска создайте инсталляцию повторно с помощью команды `exordos bootstrap`.

## Обновление

Обновления выполняются поэлементно. Ядро Exordos Core также является элементом и обновляется стандартной командой:

```bash
exordos elements update core
```

Перед обновлением сохраните версию установленного элемента, конфигурацию сети и учётные данные доступа. Перед обновлением production-контура протестируйте процедуру на отдельной локальной инсталляции и проверьте стратегию отката.
